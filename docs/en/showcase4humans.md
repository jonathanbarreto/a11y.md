# 🖼️ Showcase: Accessibility in Practice (For Humans)

Welcome! 👋

This document is the **practical showcase** of the A11Y.md standard. It was written for us, humans, who often create components following our visual intuition, but end up accidentally building barriers for those who use screen readers or navigate by keyboard.

Here, we will contrast the "way we're used to doing it" with the "accessible and semantic way."

> 💡 **Where did these examples come from?**
> The anti-patterns shown here were actually generated by using the A11Y.md project as a code reviewer of a system generated by **Gemini 3 Flash** within the **Figma Make** and **Antigravity** environments. This reinforces the importance of this project: the AI models and environments we commonly use often make basic accessibility mistakes. We are generating code faster than ever, but if we're not careful, we're also injecting more inaccessibility into the world at scale.

## 1. The "Button" That Isn't a Button (Interaction & Operability)

**Scenario:** You're creating a card on a Dashboard that acts as a shortcut to another page.

### 🔴 The Common Way (Anti-pattern)
The instinct of many developers (and even AI sometimes) is to create a `div` and attach an `onClick` to it. Visually it works, but for someone navigating by keyboard (using the `Tab` key), this `div` **simply doesn't exist**. It can't receive focus and can't be activated with the Enter key.

```tsx
// OverviewCard.tsx
<div 
  onClick={() => navigate('/components')} 
  className="glass-panel p-6 cursor-pointer hover:border-blue-500"
>
  <Puzzle className="w-5 h-5" />
  <p className="text-3xl font-bold">120</p>
  <p className="text-xs font-semibold uppercase">Components</p>
</div>
```

### ✅ The Accessible Way
If something can be clicked to trigger an action or navigation, it must be a semantic tag (`button` or `a`). By doing this, the browser automatically handles focus, Enter/Space key support, and correct screen reader announcements for free.

```tsx
// OverviewCard.tsx
<button 
  onClick={() => navigate('/components')}
  aria-label="View components list (120 indexed)"
  className="glass-panel p-6 text-left hover:border-blue-500 focus:ring-2 focus:ring-blue-500 outline-none"
>
  <Puzzle className="w-5 h-5" aria-hidden="true" />
  <p className="text-3xl font-bold">120</p>
  <p className="text-xs font-semibold uppercase">Components</p>
</button>
```

> 💡 **Remember:** Never use `div` or `span` for click interactions.

## 2. The "Fine Print" (Typography & Legibility)

**Scenario:** You need to fit a lot of information into a small card, like a "Last updated" caption.

### 🔴 The Common Way (Anti-pattern)
To let the design "breathe," we shrink the font to `10px` and lighten the color. For users with low vision or monitors with poor brightness, this text becomes completely invisible.

```tsx
// TokenList.tsx
<p className="text-[10px] text-gray-500 mt-1.5">
  Last sync 2 hours ago
</p>
```

### ✅ The Accessible Way
Keep a minimum size of `12px` (in Tailwind, this equals `text-xs`). If the color is too light, increase the contrast by making it a bit darker.

```tsx
// TokenList.tsx
<p className="text-xs text-gray-700 font-medium mt-1.5">
  Last sync 2 hours ago
</p>
```

> 💡 **Exception to the rule:** If your Design System requires `10px` for a tag/badge, you are strictly required to compensate for this with ultra-high contrast (AAA Level, 7:1) against the background.

## 3. The "Orphan" Field (Forms & Relationships)

**Scenario:** Creating a simple login screen.

### 🔴 The Common Way (Anti-pattern)
The field and the label are visually close to each other, so sighted users understand the relationship. But technically, they are disconnected. When a screen reader focuses on the input, it just says "Text box," and the user has no idea if they should type an email or a password.

```tsx
// AuthForm.tsx
<label className="block text-xs font-semibold uppercase">
  Email
</label>
<input
  type="email"
  placeholder="name@company.com"
  className="w-full px-4 py-3 border rounded-xl"
/>
```

### ✅ The Accessible Way
Link the `label` to the `input` using `htmlFor` (in React) or `for` pointing to the input's `id`. Another perk? If the user clicks the word "Email," the input automatically receives focus!

```tsx
// AuthForm.tsx
<label 
  htmlFor="auth-email" 
  className="block text-xs font-semibold uppercase"
>
  Email
</label>
<input
  id="auth-email"
  type="email"
  required
  placeholder="name@company.com"
  className="w-full px-4 py-3 border rounded-xl focus:ring-2"
/>
```

## 4. The Silent Alert (Dynamic Feedback)

**Scenario:** The user submits a form and an error occurs. A red banner pops up on the screen.

### 🔴 The Common Way (Anti-pattern)
React renders the message on the screen. But a screen reader user receives **no acoustic warning** that an error just sprouted on the interface, leaving them waiting forever for the screen to load.

```tsx
// Notification.tsx
{error && (
  <div className="p-3 bg-red-100 text-red-700 border border-red-500">
    {error}
  </div>
)}
```

### ✅ The Accessible Way
Add `role="alert"` (or `aria-live="assertive"`). This way, the exact moment the message appears in the DOM, the screen reader will interrupt whatever it's saying and warn the user about the error immediately.

```tsx
// Notification.tsx
{error && (
  <div 
    role="alert" 
    aria-live="assertive"
    className="p-3 bg-red-100 text-red-700 border border-red-500"
  >
    {error}
  </div>
)}
```

## 🚨 The Weight of Each Error

Don't know how to prioritize? Here's a quick guide on what to focus on first:

| What broke? | Severity | What does the user actually feel? |
| :--- | :--- | :--- |
| **Click on `div`** | 🔴 **CRITICAL** | *"I can't click this button using my keyboard."* |
| **Error without `role="alert"`** | 🟠 **HIGH** | *"I hit save and nothing happened, I'm stuck."* |
| **Tiny font (< 12px)** | 🟠 **HIGH** | *"My eyes hurt trying to read these dates."* |
| **Disconnected input label** | 🟠 **HIGH** | *"I focused on the field but I don't know what to type here."* |
